## Contexto

`CrmCadenceStepExecutionEntity` — entidade que representa a execução pontual de um passo de cadência — **não carrega snapshot** dos campos `template` (texto do passo) e `automatica` (flag). Hoje o `CrmProcessOverdueService.processarPasso(...)` lê esses campos lazy da cadência-mestre via relacionamento `passo → cadencia`.

**Problema latente:** se a cadência-mestre for **editada DEPOIS** que uma execução foi disparada, a execução em andamento passa a usar os valores novos. O aluno/prospect "no meio" da cadência recebe mistura de conteúdo velho (mensagens já enviadas) com conteúdo novo (próximos passos com template atualizado). Comportamento é surpresa e inconsistente com o que o usuário operador vê na UI.

É **hardening**, não bug crítico: cadências em produção raramente são editadas enquanto têm execuções ativas. Mas como o custo é baixo (2 colunas + snapshot no trigger), vale fechar.

---

## Problema / Objetivo

**Problema:** execução de cadência em andamento pode "mudar de template" se a cadência-mestre for editada. Rastreabilidade ruim.

**Objetivo:** persistir snapshot dos campos imutáveis do ponto de vista da execução (`template`, `automatica`) no momento do trigger; processador passa a ler do snapshot.

---

## Acceptance Criteria

1. **AC1** — Migração Flyway em `modulo-crm/src/main/resources/db/migration/` com nome `V{YYYYMMDDHHmm}__crm_step_execucao_snapshot.sql`:
   - Adiciona `template VARCHAR(100) NULL` em `crm_cadencia_passo_execucao`
   - Adiciona `automatica BOOLEAN NOT NULL DEFAULT false` em `crm_cadencia_passo_execucao`
2. **AC2** — `CrmCadenceStepExecutionEntity.java` atualizada com os 2 campos novos (JPA annotations + getters/setters). Snapshot populado no `@PrePersist` ou via setters explícitos na criação.
3. **AC3** — `CrmCadenceExecutionService.trigger(...)` populado os snapshots no momento da criação do step_execution, copiando de `passo.template` e `passo.automatica`.
4. **AC4** — `CrmProcessOverdueService.processarPasso(...)` passa a ler do snapshot (remove acesso lazy à cadência-mestre e o TODO inline).
5. **AC5** — Backfill SQL para step_executions existentes **em andamento** (status != TERMINADA/CANCELADA): `UPDATE crm_cadencia_passo_execucao SET template=..., automatica=... FROM passo_mestre atual`. Best-effort (valores atuais do mestre, ciente que pode diferir do momento original do trigger — documentado).
6. **AC6** — Testes:
   - `trigger_persistTemplateEAutomaticaDoMomentoAtual`
   - `processarPasso_usaSnapshotMesmoAposEdicaoDaCadencia` (edita mestre após trigger, verifica que execução continua com snapshot antigo)
   - `backfill_naoAfetaExecucoesTerminadas`

## Riscos específicos da story

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Backfill usa valor atual do mestre, não do momento real do trigger | Alta (por design) | Baixo | Documentado como "melhor esforço"; alternativa seria auditoria completa, fora do escopo |
| Constraint `automatica NOT NULL DEFAULT false` quebra rows existentes | Baixa | Médio | `DEFAULT false` cobre; testar migration em staging antes de prod |
| Divergência entre snapshot e mestre gera confusão operacional | Média | Baixo | UI da cadência deve deixar claro: "execuções em andamento usam snapshot da data X" (fora do escopo desta story, acompanhar) |

## Test plan

1. **Unit** `CrmCadenceExecutionService.trigger` — snapshot é populado com valores atuais do mestre.
2. **Unit** `CrmProcessOverdueService.processarPasso` — lê do snapshot, não do lazy-load.
3. **Integração** cenário "editou mestre depois do trigger": criar execução → editar cadência-mestre → rodar processador → verificar que comportamento segue snapshot.
4. **Migration dry-run** em staging — medir rows afetadas pelo backfill; validar que TERMINADA/CANCELADA não são tocadas.
