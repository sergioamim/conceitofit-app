## Contexto

O `academia-app` (Next.js App Router) documenta RSC como padrão em `docs/rsc-pattern.md`, porém o scan atual revela **360 arquivos com `"use client"`** (47% do projeto). Embora 412 arquivos (53%) já sejam Server Components por padrão, existem **~10 páginas críticas** que fazem data fetching inteiramente no cliente com `useQuery` + React Query.

O padrão `serverFetch()` (`src/lib/shared/server-fetch.ts`) já existe e está funcionando em **19 páginas** (ex: `backoffice/admin/leads/page.tsx`), provando que a infraestrutura está pronta.

### Problema

Páginas administrativas pesadas (900+ linhas) rodam inteiramente como Client Components, causando:
- **TTI alto** — usuário espera JS bundle inteiro antes de interagir
- **Bundle grande** — `use client` puxa React + todas as dependências para o browser
- **Lighthouse score baixo** — performance penalizada
- **SEO prejudicado** — conteúdo só renderiza no client

### Oportunidade

Migrar as 2 páginas mais pesadas da **Wave 1** usando o pattern Server Component + Client Islands já validado no projeto.

---

## Objetivo

Migrar **2 páginas** de Client Component puro para o padrão **Server Component + Client Islands**:

1. **Relatórios Contábeis** (`portal/gerencial/contabilidade/relatorios/page.tsx`)
2. **Catálogo de Segurança** (`backoffice/admin/seguranca/catalogo/page.tsx`)

**Ganho esperado:**
- TTI reduz **40-60%** nessas páginas
- Bundle inicial do browser reduz **~30-50KB** por página
- Lighthouse Performance score sobe **15-25 pontos**

### serverFetch — API existente

```typescript
// src/lib/shared/server-fetch.ts
export async function serverFetch<T>(
  path: string,
  options?: {
    query?: Record<string, string | number | boolean>;
    method?: string;
    tenantId?: string;
    next?: NextFetchRequestConfig;
    headers?: Record<string, string>;
  }
): Promise<T>;
```

**Comportamento:**
- Usa `BACKEND_PROXY_TARGET` diretamente (sem rewrite)
- Injeta `Authorization: Bearer` do cookie `academia-access-token`
- Lança `ServerFetchRequestError` em erro HTTP
- Redirect para `/login` em 401

## Notas de Implementação

### Regras a seguir

1. **NUNCA** adicionar `"use client"` no `page.tsx`
2. **SEMPRE** usar `serverFetch` para data fetching inicial no server
3. **SEMPRE** envolver Loader em `Suspense` com fallback adequado
4. **SEMPRE** exportar `dynamic = "force-dynamic"` para páginas com dados
5. Client Islands recebem dados como **props**, não fazem fetch inicial
6. Write operations (POST/PUT/DELETE) continuam via `apiRequest` nos Client Islands
7. Manter **error handling** e **loading states** equivalentes aos originais

### eslint-plugin-boundaries

O projeto usa `eslint-plugin-boundaries`. Ao criar novos componentes:
- Components em `components/` dentro do diretório da página são `internal`
- Imports de `@/components/ui/*`, `@/components/shared/*` são permitidos
- Imports de `@/lib/*` são permitidos
- Verificar que novas imports não violam boundaries existentes

### Testes

Se houver testes existentes para estas páginas:
- Atualizar imports e mocks conforme necessário
- Garantir que testes de componente funcionem com novo pattern
- Adicionar testes para novos Client Islands se não existirem
