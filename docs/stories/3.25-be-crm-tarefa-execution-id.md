## Contexto

Waves 1-3 BE do Epic 3 — Cadências CRM foram implementadas. Durante a Wave 3 (escalações), o agent de implementação levantou 4 débitos técnicos. Este é D-NEW-1.

Hoje `CrmTarefaEntity` não possui coluna `execution_id` ligando a tarefa à execução de cadência (`crm_cadencia_execucao`) que a gerou. Como consequência, a regra de escalação `TAREFA_VENCIDA` — que deveria mover a execução de etapa quando tarefas da cadência expiram sem ação — **não consegue filtrar tarefas pela execução de origem**. O `CrmProcessOverdueService.aplicarTarefaVencida(...)` fica apenas com `log.warn(...)` e um TODO inline, neutralizando uma das 3 condições de escalação desenhadas no PRD do Epic 3.

---

## Problema / Objetivo

**Problema:** sem FK `execution_id` em `crm_tarefas`, a escalação `TAREFA_VENCIDA` é um no-op. Qualquer cadência que confia em "se a tarefa não foi concluída em N dias, escale" fica quebrada.

**Objetivo:** migração aditiva + entity + service update para popular `executionId` no momento da criação da tarefa de passo, backfill best-effort para histórico, e ativação da lógica em `CrmProcessOverdueService`.

---

## Acceptance Criteria

1. **AC1** — Migração Flyway em `modulo-crm/src/main/resources/db/migration/` com nome `V{YYYYMMDDHHmm}__crm_tarefa_execution_id.sql` (convenção compacta, ver memory "Migration naming"):
   - Adiciona `execution_id UUID NULL REFERENCES crm_cadencia_execucao(id)` em `crm_tarefas`
   - Cria índice `idx_crm_tarefa_execution_id` em `crm_tarefas(execution_id)`
2. **AC2** — `CrmTarefaEntity.java` atualizada com o novo campo `executionId` (JPA annotations + getter/setter).
3. **AC3** — Método `CrmTarefaService.criarTarefaDePasso(...)` (ou equivalente chamado pelo ciclo de execução de cadência) popula `executionId` quando a tarefa nasce de um passo de execução.
4. **AC4** — Script de backfill (SQL ou Java em `modulo-crm/src/main/java/.../backfill/CrmTarefaExecutionBackfillJob.java`) que, para cada `CrmTarefaEntity` com `origem=CADENCIA` e `execution_id NULL`, tenta ligar à execução mais recente do mesmo prospect. Linhas sem match ficam com `NULL` (aceitação parcial — documentado).
5. **AC5** — `CrmProcessOverdueService.aplicarTarefaVencida(...)` filtra tarefas pela FK `execution_id` (remove o TODO + `log.warn`) e dispara o avanço de etapa conforme regra.
6. **AC6** — Testes unitários/integração:
   - `tarefaCriadaPorCadencia_temExecutionId`
   - `tarefaVencidaComExecutionId_escalacaoMoverEtapaFunciona`
   - `tarefaSemExecutionId_naoEscalaENaoLancaException` (regressão backfill parcial)

## Riscos específicos da story

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Backfill liga tarefa à execução errada (prospect teve múltiplas execuções) | Média | Médio | Critério "execução mais recente no momento da criação da tarefa"; linhas ambíguas ficam `NULL` |
| Tarefas antigas sem `execution_id` nunca escalarem | Alta | Baixo | Aceitação parcial — documentado no release note do Epic 3 |
| FK quebra em ambiente sem `crm_cadencia_execucao` populado | Baixa | Alto | Migration usa `ON DELETE SET NULL` + coluna `NULL`able |

## Test plan

1. **Unit** `CrmTarefaService` — `executionId` é setado quando `origem=CADENCIA` e `passoExecucao` presente.
2. **Unit** `CrmProcessOverdueService.aplicarTarefaVencida` com mocks — escalação dispara apenas quando tarefa ligada à execução está vencida.
3. **Integração** `@SpringBootTest` com Testcontainers — fluxo E2E: criar execução → criar tarefa → vencer tarefa → rodar scheduler → verificar etapa movida.
4. **Regressão** tarefas criadas manualmente (origem ≠ `CADENCIA`) continuam ignoradas pela escalação.
5. **Backfill dry-run** em staging — medir taxa de match (esperado: >70% em tenants com <6 meses de histórico).
